# 邮件处理方案

本文档详细说明 Apple Order Manager 的邮件处理机制、技术方案和实现细节。

---

## 一、方案概述

### 核心机制

| 维度 | 方案 |
|------|------|
| **实时处理** | IMAP IDLE 推送（< 5 秒响应） |
| **搜索策略** | `['UNSEEN']` 未读邮件 |
| **并发控制** | `isProcessing` 标志位（单线程串行） |
| **标记已读** | 选择性标记（订单邮件成功后标记） |
| **定时回溯** | 每 10 分钟回溯 30 分钟 |
| **启动回溯** | 回溯 24 小时 |

### 关键决策点说明

| 🔍 决策点 | 判断条件 | ✅ YES → | ❌ NO → |
|----------|---------|---------|---------|
| **isProcessing** | 当前是否正在处理邮件 | 直接返回（避免并发）| 开始处理 |
| **找到邮件** | 搜索结果是否为空 | 获取 UID 列表 | 返回 IDLE 状态 |
| **UID 已处理** | `email_logs` 表存在该 UID | 跳过该邮件 | 下载元数据 |
| **是订单邮件** | 发件人 + 主题匹配 | 下载完整内容 | 跳过（不标记已读）|
| **解析成功** | 能提取订单号/产品等 | 检查订单号 | 跳过（不标记已读，下次重试）|
| **订单已存在** | `orders` 表存在该订单号 | 跳过（标记已读）| 保存到数据库 |
| **再次检查未读** | 处理完后是否还有未读 | 立即再次处理 | 返回 IDLE 状态 |

## 三、选择性标记已读策略

### 标记规则

| 邮件类型 | 处理结果 | 是否标记已读 | 原因 |
|---------|---------|-------------|------|
| 订单邮件 | 处理成功 | ✅ 标记 | 避免重复搜索，性能最优 |
| 订单邮件 | 解析失败 | ❌ 不标记 | 保持未读，下次重试 |
| 订单邮件 | 订单已存在 | ✅ 标记 | 虽然跳过，但已处理过 |
| 非订单邮件 | - | ❌ 不标记 | 不干预用户邮箱 |

### 技术实现

```javascript
// 成功处理订单后标记已读
await saveOrderToDatabase(orderData);
await recordEmailLog(emailUid);

// 调用 IMAP 标记已读
imapConnection.addFlags(emailUid, '\\Seen', (err) => {
  if (err) {
    logger.warn('标记邮件已读失败', { emailUid, error: err.message });
  } else {
    logger.info('邮件已标记已读', { emailUid });
  }
});
```

## 四、三种触发场景

### 场景对比

| 触发方式 | 搜索条件 | 触发频率 | 目的 | 并发控制 |
|---------|---------|---------|------|---------|
| **IDLE 实时** | `['UNSEEN']` | 新邮件到达（秒级） | 实时处理 | isProcessing 标志位 |
| **定时回溯** | `['SINCE', 30分钟前]` | 每 10 分钟 | 容错补漏 | isProcessing 标志位 |
| **服务启动** | `['SINCE', 24小时前]` | 每次启动 | 补漏 | 无 |

### 定时回溯逻辑

```javascript
// 每 10 分钟执行一次
cron.schedule('*/10 * * * *', async () => {
  // 等待当前处理完成
  while (isProcessing) {
    await sleep(1000);
  }
  
  isProcessing = true;
  
  // 搜索最近 30 分钟的邮件
  const sinceDate = new Date(Date.now() - 30 * 60 * 1000);
  const results = await imapConnection.search(['SINCE', sinceDate]);
  
  // 处理逻辑与 IDLE 相同
  // UID 去重会自动跳过已处理的邮件
  
  isProcessing = false;
});
```

## 五、去重机制（三层防护）

### 去重层级

| 层级 | 检查时机 | 检查方式 | 拦截效果 |
|------|---------|---------|---------|
| **第 1 层** | 下载邮件前 | `email_logs` 表 UID 查询 | 同一封邮件只处理 1 次 |
| **第 2 层** | 保存订单前 | `orders` 表订单号查询 | 不同邮件相同订单只录入 1 次 |
| **第 3 层** | 数据库插入时 | 订单号唯一索引 | 兜底防止并发冲突 |

### 第 1 层：UID 去重

```javascript
// 批量查询（性能优化）
const existingUids = await EmailLog.findAll({
  where: { email_uid: { [Op.in]: uidList } },
  attributes: ['email_uid']
});

const existingUidSet = new Set(existingUids.map(log => log.email_uid));

for (const uid of uidList) {
  if (existingUidSet.has(uid)) {
    logger.debug('UID 已处理，跳过', { uid });
    continue;
  }
  // 处理邮件...
}
```

## 六、性能指标

### 场景测试

**场景 1：单封邮件**
- 响应时间：**2-5 秒** ✅

**场景 2：1 分钟 100 封邮件**
- 首封响应：**立即（0 秒）** ✅
- 末封响应：**40-60 秒**（串行处理）✅
- IMAP 搜索次数：**1-2 次** ✅

**场景 3：每 2 秒 1 封邮件，持续 5 分钟**
- 首封响应：**立即** ✅
- 后续邮件：**串行处理，无延迟累加** ✅
- 不会出现防抖导致的 5 分钟延迟 ✅

**场景 4：1000 封已处理邮件 + 1 封新邮件**
- 搜索 `['UNSEEN']` 只返回 1 封新邮件（已处理的已标记已读）
- 响应时间：**2-5 秒** ✅
- 性能最优（无需 UID 去重）✅

## 七、邮件过滤规则

### 过滤层级

| 过滤层 | 规则 | 不匹配时 |
|-------|------|---------|
| **UID 去重** | `email_logs` 表不存在该 UID | 跳过该邮件 |
| **元数据过滤** | 发件人包含 `lanu.cn` 或 `noreply` <br> **且** 主题包含 `NULL`/`预订助手`/`预订成功` | 跳过该邮件（不标记已读）|
| **内容解析** | 能提取订单号（W+9位）、Apple ID、产品信息 | 跳过该邮件（不标记已读）|
| **订单去重** | `orders` 表不存在该订单号 | 跳过该邮件（标记已读）|

```bash
# IMAP 配置
IMAP_HOST=imap.qq.com
IMAP_PORT=993

IMAP_PASSWORD=your_auth_code
```

|------|-----|------|
| `IMAP_SEARCH` | `['UNSEEN']` | 搜索未读邮件 |
| `keepalive` | `true` | 保持 IMAP 长连接 |
| `定时回溯频率` | 10 分钟 | 业务需求 |
| `定时回溯范围` | 30 分钟 | 业务需求 |
| `启动回溯范围` | 24 小时 | 补漏 |
| `断线重连延迟` | 30 秒 | 容错 |

## 九、异常处理

### 异常类型与处理

| 异常类型 | 处理方式 | 影响范围 |
|---------|---------|---------|
| **网络断线** | 30 秒后自动重连 | 临时中断，重连后恢复 |
| **UID 已存在** | 跳过，记录日志 | 单封邮件 |
| **非订单邮件** | 跳过，记录日志 | 单封邮件 |
| **解析失败** | 跳过，记录错误日志，保持未读 | 单封邮件，下次重试 |
| **订单已存在** | 跳过，记录日志，标记已读 | 单封邮件 |
| **数据库异常** | 事务回滚，记录错误，保持未读 | 单封邮件，下次重试 |
| **标记已读失败** | 记录警告日志，不影响主流程 | 下次会重复处理（UID 去重会拦截）|

## 十、技术亮点

### 1. 选择性标记已读（核心优化）

**优势**：
- ✅ 性能最优：搜索 `['UNSEEN']` 自动过滤已处理邮件
- ✅ 支持重试：失败邮件保持未读，下次自动重试
- ✅ 不干扰用户：非订单邮件保持原状

### 2. isProcessing 标志位（并发控制）

**优势**：
- ✅ 实现简单：单个布尔变量
- ✅ 避免并发：正在处理时直接返回
- ✅ 无延迟：不使用防抖，立即响应

### 3. 三种触发互补（容错保障）

**优势**：
- ✅ IDLE 实时处理（秒级响应）
- ✅ 定时回溯（10 分钟容错窗口）
- ✅ 启动回溯（24 小时补漏）

| 前提 | 说明 | 重要性 |

## 十二、方案演进历史

### 版本 1.0（初始方案）
- 搜索 `['UNSEEN']`
- 无防抖
- 不标记已读
- ❌ 问题：每次都搜索所有未读邮件，UID 去重开销

### 版本 2.0（防抖方案）
- 回溯 5 分钟
- 3 秒防抖
- 不标记已读
- ❌ 问题：高频场景下延迟 5 分钟+

### 版本 3.0（当前方案）✅
- 搜索 `['UNSEEN']`
- isProcessing 标志位
- **选择性标记已读**
- ✅ 性能最优 + 支持重试 + 实时响应
